Stage 5 - End to End QA

Stage Objectives

Execute the continuous testing pipeline with all the pre-defined quality gates.

In prior stages, you have seen how tests are required to verify all facets of the application viz. – functional, performance, security, accessibility etc. You have also seen the need for quality assurance for not just the UI level, but also at a component level viz. – code, build, deployment, data & interfacing services.

In this stage, we focus on all these facets of the application and cover reporting and dashboard at an end-to-end level.

Pipeline Overview

In this fifth and final stage, you will execute all the same tests executed as the part of previous stages, but in a continuous testing pipeline from Azure Repos to final reports. Additionally, in this stage you will also read about new-age QA practices like AI driven QA that are increasingly becoming critical for modern-day testing in DevOps.

End-to-End Quality Assurance flow:

End-to-End Quality Assurance Pipeline consists of:

  • Build Pipeline
  • Release Pipeline

The End-to-End-QA Build pipeline consist of 3 stages:

Build Pipeline Front End

This stage is responsible for deployment of front-end and consists of developing quality gate for unit test and code coverage, then building and pushing the image to Azure Container Registry (ACR), followed by deployment with virtual service to AKS and then BVT-UI Test (Build Verification Testing).

Build Pipeline Backend

This stage is responsible for deployment of back-end and consists of developing quality gate for unit test and code coverage then building and pushing MySQL image to ACR.

Functional Test Build and Push Image

Contains source code for all the subsequent Functional Test stages i.e. Smoke, API, UI and RWD testing types and the code for experience assurance testing types, namely – performance, security and accessibility and includes building and pushing image to ACR and publishing the artifacts to the release pipeline.

The Deployment Functional Experience Release Pipeline consist of following stages:

Deployment

On successful build, AWS Systems Manager is leveraged to deploy back-end and front-end components of the application to Azure instance.

Smoke Test

Runs smoke test to qualify the build for further stages of testing.

Functional Assurance

Evaluates functional behaviors i.e. “what” the system should do (or) if the application / system functions perform as expected, by executing a comprehensive set of APIs, UI and RWD tests as parallel Code Build jobs.

Experience Assurance

Evaluates non-functional behaviors i.e. “How” the system should perform its functions by executing security and accessibility tests followed by performance as sequential Code Build jobs.

Debugging & Re-Excution

To understand in detail why the failure happened, please check the error message by going to “Tests” tab of the UI Tests. Refer screenshot below for details:

You will be able to see the report with the error:

Click on the failed Test Case to read the error message from the Debug tab.

org.openqa.selenium.NoSuchElementException: no such element: Unable to locate element: "method":"xpath","selector":"//span[contains(text(),’Incorrect Password’)]"

Such errors indicate that the automation script is unable to locate the object on the screen as properties of that UI object have changed since the last build. This is a very common error and calls for script maintenance

To fix this issue, navigate to the Azure Repositories of your project, find the repository “azure_workshop_functional_assurance” and edit the file ShopLoginPage.java azure_workshop_functional_assurance/src/test/java/pages/ShopLoginPage.java

Refer the screenshot as shown below for details:

You will notice at line# 61 as: @FindBy(xpath = “//span[contains(text(),’Incorrect Password’)]”).

Change the ’Incorrect Password’ to ‘Customer not found with email’, provide the required details like comment and click ‘Commit’. Refer screenshot below for more details:
